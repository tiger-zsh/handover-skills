# 交接审计 Checklist

> 这是一份**共享 checklist 模板**，用于评估"新人是否真能接手这个项目"。
>
> 当前阶段（阶段 1）`handover-audit` 不是独立 skill。你可以在以下场景手动使用本模板：
>
> - 离职 / 转岗前自检
> - 评估一个项目的交接完成度
> - 完成 onboarding / runbook / architecture 后做收尾检查
>
> 等阶段 3 评估后，再决定是否把这套 checklist 升级为独立 skill。

---

## 使用方法

把这份模板复制到目标项目的 `docs/handover-checklist.md`，逐条勾选。**不能勾选的项**就是交接缺口。

每项标记 `[x] / [ ] / [N/A]`，并在末尾汇总缺口。

---

## 一、入门链路（onboarding 覆盖）

- [ ] 项目 `docs/onboarding.md` 存在且最近一次更新在 3 个月内
- [ ] 新人能按文档完成本地启动（实测过）
- [ ] 新人能按文档跑通主要测试（实测过）
- [ ] `docs/onboarding.md` 列出 3-5 个核心入口
- [ ] `docs/onboarding.md` 列出第一个低风险练手任务
- [ ] 文档中没有遗留的 `[待确认]` 项（或留下的项不影响接手）

## 二、架构理解（architecture-map 覆盖）

- [ ] `docs/architecture.md` 存在
- [ ] 模块边界清晰
- [ ] 数据流（请求路径、消息流）有描述
- [ ] 外部依赖（数据库、队列、第三方服务）有列出
- [ ] 关键设计取舍有说明（"为什么这样拆"）
- [ ] 已知技术债有记录

## 三、生产运维（runbook 覆盖）

- [ ] `docs/runbook.md` 存在
- [ ] 发布流程明确（每一步带命令）
- [ ] 回滚路径明确（每个高风险操作都有回滚）
- [ ] 备份策略和位置明确
- [ ] 主要告警渠道明确（去哪里看告警）
- [ ] secret / credential 存放位置明确（但**不**写明文）
- [ ] 数据库恢复流程演练过

> ⚠️ runbook 部分凡是 `[待确认]` 的项，**接手前必须清零**，不能带着模糊上线。

## 四、故障剧本（sop 覆盖）

- [ ] `docs/sop/INDEX.md` 存在
- [ ] 过去 6 个月的非平凡故障都有 SOP 条目
- [ ] 关键词命中率：随机挑 3 个历史故障的关键词，能在 INDEX 里搜到

## 五、权限 / 账号 / 密钥

- [ ] GitHub / GitLab 仓库权限交接路径明确
- [ ] 生产数据库账号交接路径明确
- [ ] 云厂商账号（AWS / GCP / 阿里云等）交接路径明确
- [ ] 域名 / 证书所有权交接路径明确
- [ ] 第三方服务（支付、邮件、监控等）账号交接路径明确
- [ ] CI/CD 的 secret 谁能维护明确

## 六、单点知识风险

- [ ] 没有"只有某个人知道"的模块（或者已经显式记录在 owner 表中）
- [ ] 历史决策的关键讨论有归档（chat 记录、PR 讨论、设计文档）
- [ ] 关键外部联系人（合作方、客服对接人）有列出

## 七、高风险区

- [ ] 已列出哪些操作"不要随便动"（生产数据修复、定时任务、数据迁移）
- [ ] 关键定时任务的执行时机、影响、关闭方式明确
- [ ] 数据库 schema / 数据迁移流程明确（带回滚）

---

## 缺口汇总

把上面没勾选的项汇总到这里，按严重程度分级：

### 🔴 阻塞性（接手前必须清零）

- {{...}}

### 🟡 重要（接手后两周内补齐）

- {{...}}

### 🟢 改进项（有空再补）

- {{...}}

---

## 给接手人的话

- 第一周以**读 + 跑通本地 + 完成练手任务**为主。
- 不要在前两周做生产操作，先把 runbook 全部 `[待确认]` 项清掉。
- 遇到错误先查 `docs/sop/INDEX.md`，没命中再问。
- 每解决一个非平凡问题，记得回写到 SOP。
