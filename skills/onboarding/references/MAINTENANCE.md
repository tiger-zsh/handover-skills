# onboarding 文档维护

`docs/onboarding.md` 一次生成完就过期是常态。本文档说明**何时**应该重新跑 onboarding skill 来更新它，以及更新时的行为约束。

## 何时建议更新

当项目里发生以下变化时，建议提示用户跑 `/onboarding` 更新：

| 变化 | 影响章节 |
|---|---|
| `package.json` / `pyproject.toml` / `go.mod` / `Cargo.toml` 等依赖清单变更 | 环境准备、安装步骤 |
| 启动脚本变更（`scripts/start*`、`Makefile`、`justfile`、`docker-compose.yml`） | 本地启动命令 |
| 测试入口变更（测试命令、CI 配置中的测试 job） | 测试命令 |
| 主要目录结构调整、核心模块改名或重组 | 核心入口列表 |
| `.env.example` 增加或删除关键变量 | 环境变量、本地启动 |
| `README.md` / `CONTRIBUTING.md` 大幅改写 | 整体内容可能需要复审 |

不强制每次都更新——小修小补不必触发，整体方向变了才需要。

## 更新行为约束

进入"更新模式"时（即目标文件已存在）：

1. **不要全量重写**。读取现有 `docs/onboarding.md`，只改差异部分。
2. **保留所有带 `[来源：用户确认]` 标记的段落**——这是用户在前一次生成时明确确认过的，除非用户本次明确说要修改，否则保持原样。
3. **重新探测时如果发现某条"用户确认"的事实和代码冲突**：不要直接覆盖，而是把冲突报告给用户（"代码里看到 X，但文档里记录的是 Y，以哪个为准？"），由用户决定。
4. **新增内容默认标 `[来源：待确认]`**，按主流程逐节确认后再升级标记。
5. **删除内容前必须确认**——避免误删用户手工补充的段落。

## 不属于 onboarding 的内容

如果在更新时发现现有 `onboarding.md` 含有以下内容，提示用户拆分出去，**不要**自己直接迁移（迁移由对应 skill 负责）：

- 生产部署 / 回滚 / 备份 / 告警 → `runbook`（阶段 2）
- 完整模块边界 / 数据流图 / 设计取舍 → `architecture-map`（阶段 2）
- 历史故障排查记录 → `sop`

明确告诉用户："这部分归 X，建议等对应 skill 上线后迁移过去。"目前阶段先保留在 onboarding 内也可以，但加注释标明"将来归 X"。
